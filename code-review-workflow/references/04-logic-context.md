# 04 — 逻辑背景获取策略

逻辑 BUG 审查的成败在于**有没有业务背景**。同样一段代码，知道业务规则就能看出问题，不知道就只能审技术层面。

本文档定义**三步获取策略**，按优先级递减使用。前一步拿不到，用下一步；都拿不到，降级处理。

---

## 三步策略总览

```
步骤1：扫项目文档目录（doc/ docs/ README 等）
    ↓ 没找到相关业务文档
步骤2：委托 AI agent 探测已安装的知识库技能
    ↓ 没有知识库技能 / 技能中没有相关内容
步骤3：从代码本身推断业务规则
    ↓ 推断不出来
降级：逻辑层只标注"疑似"，列出待确认点
```

---

## 步骤 1：扫项目文档目录

### 找什么

扫描项目根目录及常见文档位置，寻找业务说明类文档：

| 位置 | 找什么 |
|------|--------|
| `doc/` `docs/` `documentation/` | 项目文档目录 |
| `README.md` `readme.md` | 项目说明 |
| `*.md`（根目录及一级子目录） | 设计文档、需求文档、变更说明 |
| `wiki/` `设计文档/` `需求文档/` | 中���命名的文档目录 |
| `CHANGELOG.md` `CHANGES.md` | 变更记录（可推断业务演进） |

### 怎么扫

用 Glob 搜索这些路径下的 Markdown / 文档文件：

```
Glob pattern: doc/**/*.md
Glob pattern: docs/**/*.md
Glob pattern: **/*.md （根目录及一级子目录，限制深度）
```

找到后，**优先读与审查代码相关的文档**：
- 文件名/标题含审查模块关键词的优先
- 业务流程图、状态机图、接口文档优先
- 纯技术选型/部署文档（如"用什么框架"）跳过

### 提取什么

从文档中提取**业务规则**，用于逻辑 BUG 审查：

- **数据流转规则**：数据从哪来、经过什么处理、到哪去
- **状态定义与流转**：有哪些状态、合法的状态转移路径
- **校验规则**：业务约束（如"金额不能为负""审批需要两级"）
- **边界与异常场景**：业务定义的边界条件、异常处理预期
- **权限模型**：谁能做什么、数据隔离规则

### 记录背景来源

审逻辑 BUG 时，**报告中必须注明业务规则从哪来**：

```markdown
📌 **背景来源**：项目 `docs/order-flow.md` 中"订单状态流转"章节
```

> 如果文档与代码冲突（文档说 A，代码做 B），以代码为准但标注差异，因为这往往就是逻辑 BUG 或文档过时。

---

## 步骤 2：委托 AI agent 探测已安装的知识库技能

> ⚠️ **核心设计**：本步骤**不硬编码**调用某个特定技能（如 taihong-knowledge），而是让 AI agent 自行判断当前环境有哪些已安装的技能可用。

### 为什么要委托 AI agent

不同项目装的知识库技能不同：
- 台宏 IMP 项目可能装了 `taihong-knowledge`
- 其他项目可能装了别的知识库技能
- 有的项目什么都没装

**硬编码"调 taihong-knowledge"会导致技能在其他项目失效**。正确做法是让 AI agent 自己探测。

### 怎么委托

审查逻辑 BUG 前，如果步骤 1（扫文档）没找到足够的业务背景，按以下方式委托 AI agent：

**方式 A：用 Agent 工具探测（推荐）**

启动一个 Explore 类型的 agent，让它去搜索项目相关的知识库技能或文档：

```
Agent（Explore 类型）：
"我正在审查 [某模块] 的代码，需要了解项目的业务背景。
请帮我查找：
1. 当前环境中是否安装了项目知识库相关的技能？检查可用技能列表中是否有知识库/项目背景类的技能。
2. 如果有，调用它查询 [审查模块] 相关的业务规则、数据流转、状态定义。
3. 同时检查项目目录中 doc/docs/ 等位置是否有业务文档。
汇报你找到的业务背景信息。"
```

**方式 B：AI 自行判断可用技能**

AI agent 审查时，检查自身可用技能列表（系统提示中的 `<available_skills>`），寻找匹配的知识库技能：

- 技能描述含"知识库""项目背景""业务逻辑""检索项目"等关键词的
- 技能描述含项目名或领域名的（如"台宏""IMP"）

找到后，按该技能的加载规则使用。

### 拿到背景后

将知识库技能提供的业务规则，与代码实现对照审查。同样**记录背景来源**：

```markdown
📌 **背景来源**：通过 [技能名] 知识库技能查询，依据 [具体文档/条目]
```

### 探测不到怎么办

如果 AI agent 报告"没有找到知识库技能"或"技能中无相关内容"，进入步骤 3。

---

## 步骤 3：从代码本身推断业务规则

没有文档、没有知识库时，从代码中**推断**业务意图。这准确度最低，但聊胜于无。

### 推断方法

| 线索 | 能推断什么 |
|------|-----------|
| **命名** | `createOrder`→创建订单；`refundAmount`→退款金额；`status=3`→看枚举或注释推状态含义 |
| **注释** | 方法注释、行内注释往往说明业务意图 |
| **调用链** | 顺着调用关系看数据怎么流动、谁调谁 |
| **数据库 schema** | 表名、字段名、注释（如有）、枚举字段的可选值 |
| **测试代码** | 测试用例的输入输出反映预期行为（如果有测试的话） |
| **相似模块** | 项目中类似功能的实现方式，可参照推断 |
| **前端代码** | 前端的校验提示、按钮文案往往暴露业务规则 |

### 推断的局限

推断出来的业务规则**不确定**。审出来的逻辑 BUG 必须标注"疑似"：

```markdown
**❓ [疑似问题标题]**
- 📝 **描述**：基于代码推断，此处逻辑可能是 [X]，实际行为是 [Y]，存在 [Z] 风险。
- ⚠️ **待确认**：业务规则我是从 [命名/注释/调用链] 推断的，不确定是否准确。请确认实际业务规则。
- 📌 **背景来源**：纯代码推断（无业务文档/知识库）。
```

---

## 降级处理：背景完全不足时

三步都走完仍然拿不到业务背景（罕见，通常至少步骤 3 能推断一些）：

1. **逻辑 BUG 审查只标注"疑似"或"未审查"**，不硬下结论。
2. **报告头部明确说明背景不足**：

```markdown
## 审查概要
- **业务背景来源**：⚠️ 无业务文档、无知识库技能、代码推断有限
- **逻辑 BUG 审查状态**：降级模式，仅标注疑似问题，未做确认性审查
```

3. **在疑似问题中列出"如果有以下业务规则，这里可能有 BUG"**：

```markdown
**❓ 待业务确认后方可判断**
- 如果业务规则是"退款不能超过原价"，当前代码未做此校验，可能有超额退款风险。
- 如果业务规则是"订单取消后库存必须回滚"，当前代码未看到回滚逻辑，可能有库存不一致风险。
- ⚠️ 以上均为假设，请业务方确认实际规则后复核。
```

4. **建议用户补文档**：可推荐使用 `code-to-doc-workflow` 技能从代码逆向生成文档，下次审查就有背景了。

---

## 背景充足度自评

每次审查逻辑 BUG 前，自评背景充足度，决定审查深度：

| 充足度 | 判定 | 审查策略 |
|--------|------|---------|
| 🟢 充足 | 有业务文档/知识库明确描述了相关规则 | 正常审查，问题可下结论 |
| 🟡 部分 | 有部分文档或能从代码较可靠推断 | 审查但标注"依据来源"，关键处标疑似 |
| 🔴 不足 | 无文档、无知识库、代码也推断不出 | 只标疑似，列待确认清单，不硬下结论 |

> 这个自评结果写进报告概要的"业务背景来源"字段，让用户知道你的逻辑审查有多可信。
