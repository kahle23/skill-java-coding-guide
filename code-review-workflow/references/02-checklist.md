# 02 — 审查清单

审查清单是**防止漏审**的工具。审查时对着清单逐项过，不凭印象。

清单分四层：
1. **通用清单** — 所有语言适用（代码风格/可读性/复杂度）
2. **代码 BUG 专项** — 技术缺陷，与业务无关，所有语言适用
3. **逻辑 BUG 专项** — 业务缺陷，需要业务背景
4. **语言专项** — Java / Vue(前端) / Python 各自特有

---

## 第一层：通用清单

### 代码风格与可读性

| # | 检查项 | 说明 |
|---|--------|------|
| G1 | 命名是否达意 | 变量/方法/类名是否准确表达意图。`data`/`info`/`temp` 这种含糊名要质疑。 |
| G2 | 方法是否过长 | 超过 50 行的方法考虑拆分（除纯映射/配置类）。 |
| G3 | 嵌套是否过深 | 超过 3 层 if/for 嵌套考虑用卫语句/提前 return 提取。 |
| G4 | 注释是否有用 | 注释应解释"为什么"而非"做什么"。代码已自解释的废话注释建议删。 |
| G5 | 重复代码 | 相同逻辑出现 3+ 处，建议抽取。 |
| G6 | 魔法数字/字符串 | 硬编码的数字/字符串建议提为常量。 |

> 通用层问题通常是**改进建议**级别，不是致命/严重。

## 第二层：代码 BUG 专项（⚠️技术缺陷，与业务无关）

### C1：空指针 / 空引用

| 检查点 | 典型场景 |
|--------|---------|
| 方法返回值未判空就使用 | `list.get(0)` 前没判空 / `map.get(key)` 直接调方法 |
| 链式调用中间可能为 null | `obj.getA().getB().getC()` 任一层 null 就炸 |
| 外部输入未校验 | 接口参数 / 配置值 / 文件内容 直接使用 |
| 集合元素未判空 | 遍历集合时元素可能为 null |

### C2：资源泄漏

| 检查点 | 典型场景 |
|--------|---------|
| 流/连接/文件未关闭 | IO 流、DB 连接、网络连接异常路径未关闭 |
| 未用 try-with-resources / with / finally | Java 未用 try-with-resources；Python 未用 with |
| 提前 return / break 漏关闭 | 循环或条件分支中提前退出，跳过了关闭逻辑 |

### C3：并发问题

| 检查点 | 典型场景 |
|--------|---------|
| 共享可变状态无同步 | 多线程读写同一 HashMap / List / 变量 |
| 检查-执行竞态（TOCTOU） | `if (map.get(k) == null) map.put(k, v)` 非原子 |
| 锁粒度不当 | 锁太大（性能）或太小（没保护到） |
| ThreadLocal 未清理 | 线程池复用线程，ThreadLocal 数据泄漏 |

### C4：安全漏洞

| 检查点 | 典型场景 |
|--------|---------|
| SQL 注入 | 字符串拼接 SQL，未用参数化查询 |
| XSS | 前端渲染未转义用户输入（v-html / innerHTML） |
| 敏感信息泄漏 | 密码/密钥日志打印、错误信息回显内部细节 |
| 路径遍历 | 文件操作拼接用户输入的路径 |

### C5：异常处理

| 检查点 | 典型场景 |
|--------|---------|
| 异常吞没 | `catch (Exception e) {}` 空捕获，问题被隐藏 |
| 捕获过宽 | `catch (Exception)` 吞掉所有异常，掩盖特定问题 |
| 异常类型错误 | 业务错误用 RuntimeException、编程错误用受检异常 |
| finally 抛异常 | finally 中再抛异常会覆盖 try 中的原始异常 |

### C6：类型安全

| 检查点 | 典型场景 |
|--------|---------|
| 隐式类型转换 | JS `==` 比较、Java 自动装箱拆箱、Python 动态类型混用 |
| 泛型擦除陷阱 | Java 运行时泛型类型丢失，强转不安全 |
| 可变默认参数 | Python 可变默认参数在函数定义时只创建一次 |

### C7：性能隐患

| 检查点 | 典型场景 |
|--------|---------|
| N+1 查询 | 循环内查数据库，应批量查 |
| 循环内建对象/执行重复计算 | 循环内 new 对象、循环内做不变的字符串拼接 |
| 全表扫描 / 缺索引 | 查询条件无索引、LIKE '%xx%' 前导通配 |
| 集合选型不当 | 频繁 contains 用 List 而非 Set；频繁插入头部用 ArrayList |

## 第三层：逻辑 BUG 专项（⚠️业务缺陷，需要业务背景）

> ⚠️ **本层审查前必须先获取业务背景**（见 04-logic-context.md）。无背景时只能标"疑似"。

### L1：边界条件

| 检查点 | 典型场景 |
|--------|---------|
| 空值/空集合处理 | 列表为空时的逻辑分支是否覆盖 |
| 极值处理 | 数量为 0 / 负数 / 最大值时的行为 |
| 越界检查 | 索引、分页参数的边界 |
| 日期/时间边界 | 跨天/跨月/跨年、时区、夏令时 |

### L2：状态流转

| 检查点 | 典型场景 |
|--------|---------|
| 状态机完备性 | 所有状态都有对应处理？非法状态转移是否拦截？ |
| 中间态处理 | 异步操作进行中的状态是否正确 |
| 状态回退 | 失败后状态是否正确回滚 |

### L3：幂等性 / 重复提交

| 检查点 | 典型场景 |
|--------|---------|
| 重复提交防护 | 网络重试/用户双击导致重复执行 |
| 幂等键设计 | 唯一约束 / 幂等 token 是否到位 |

### L4：金额 / 数量计算

| 检查点 | 典型场景 |
|--------|---------|
| 浮点精度 | 用 double/float 算钱，应 BigDecimal / 分整数 |
| 累加误差 | 多次浮点运算累积误差 |
| 货币单位混淆 | 元/分混用 |
| 四舍五入策略 | ROUND_HALF_UP vs DOWN vs BANKER |

### L5：权限 / 数据隔离

| 检查点 | 典型场景 |
|--------|---------|
| 越权访问 | 接口未校验当前用户是否有权操作该数据 |
| 多租户隔离 | 查询未带 tenant_id，跨租户数据泄漏 |
| 数据归属校验 | 操作前未验证数据属于当前用户/组织 |

### L6：数据一致性

| 检查点 | 典型场景 |
|--------|---------|
| 事务边界 | 多表/多步操作未在同一事务，部分失败导致不一致 |
| 并发更新覆盖 | 乐观锁/悲观锁缺失，并发更新互相覆盖 |
| 缓存与 DB 不一致 | 更新 DB 后缓存未失效或顺序错误 |

## 第四层：语言专项

### Java 专项

| # | 检查项 | 说明 |
|---|--------|------|
| J1 | equals / hashCode 契约 | 重写 equals 必须重写 hashCode；用 == 比对象而非 equals |
| J2 | 集合操作陷阱 | `subList` 视图、`asList` 不可变、修改集合时 ConcurrentModificationException |
| J3 | 日期时间 API | Date/Calendar 线程不安全，应 java.time；SimpleDateFormat 线程不安全 |
| J4 | 序列化陷阱 | Serializable 未显式 serialVersionUID；transient 字段反序列化后 null |
| J5 | 反射性能 | 循环内反射调用应缓存 Method 对象 |
| J6 | 注解误用 | @Autowired 字段注入 vs 构造器注入；@Transactional 自调用不生效 |
| J7 | Optional 误用 | Optional 作为字段/参数；直接 get() 不判 isPresent |

### Vue / 前端专项

| # | 检查项 | 说明 |
|---|--------|------|
| V1 | 响应式陷阱 | Vue2: `this.$set` / `Vue.set` 动态加属性；Vue3: `reactive` 解构丢响应式 |
| V2 | v-for key | 必须有稳定唯一 key，禁止用 index 做 key |
| V3 | 异步生命周期 | created/mounted 中发请求，数据未到就操作 → 空值错误 |
| V4 | watch 误用 | deep watch 性能问题；immediate 时机；watch 监听 ref/reactive 区别 |
| V5 | 组件通信 | props 单向流、避免直接改 props、$emit 命名规范 |
| V6 | 内存泄漏 | 定时器/事件监听/全局订阅未在 beforeUnmount/beforeDestroy 清理 |
| V7 | XSS | v-html 渲染用户输入、innerHTML 拼接 |
| V8 | 异步竞态 | 多次请求结果乱序到达，旧数据覆盖新数据 |

### Python 专项

| # | 检查项 | 说明 |
|---|--------|------|
| P1 | 可变默认参数 | `def f(items=[])` 默认值在定义时只创建一次，跨调用共享 |
| P2 | 闭包变量绑定 | lambda / 闭包捕获循环变量，实际取最后一次值 |
| P3 | 浅拷贝陷阱 | `copy.copy` 只拷贝一层，嵌套结构共享引用 |
| P4 | 异常过宽 | `except:` 或 `except Exception:` 吞掉所有 |
| P5 | 全局状态 | 模块级可变变量在并发/多次调用中被意外修改 |
| P6 | 类型注解一致性 | 注解标了 Optional 但实际未处理 None；运行时类型与注解不符 |
| P7 | 生成器只消费一次 | 生成器遍历后耗尽，再次遍历为空 |
| P8 | 字符串编码 | Python2/3 混用、默认编码非 UTF-8 环境下的乱码 |

---

## 使用方法

1. **每次审查，清单至少过到第二层（代码 BUG 专项）**，这是与业务无关的硬技术问题。
2. **第三层（逻辑 BUG）视背景决定**：有背景则逐项过；无背景则标注"疑似"或"未审查"。
3. **第四层（语言专项）按代码语言选**：Java 代码过 J 项，Vue 过 V 项，Python 过 P 项。
4. **清单是防漏工具，不是教条**：清单没列但发现的问题也要报告。清单帮助系统化，不限制发现。
