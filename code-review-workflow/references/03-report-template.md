# 03 — 报告模板

审查报告是技能的**唯一交付物**。报告质量决定审查价值。核心原则：**分级清晰、每条可执行、对事不对人**。

---

## 报告整体结构

```
# 代码审查报告

## 审查概要
（审了什么、用了什么背景、发现几类问题各多少）

## 🔴 致命问题（N 项）
（必须修，否则上线出事 / 数据损坏 / 安全漏洞）

## 🟠 严重问题（N 项）
（强烈建议修，有隐患，特定条件下会出事）

## 🟡 改进建议（N 项）
（可选优化，提升可读性/可维护性/性能）

## 🟢 值得肯定（N 项）
（好的实践，继续保持）

## ❓ 疑似问题（N 项）
（背景不足无法确认，列出待确认点）

## 审查范围说明
（本次审查覆盖了什么、没覆盖什么）
```

> 某一级别没有问题时，写"无"或省略该节。疑似问题为空时也要说明"逻辑背景充足，无疑似问题"。

---

## 四级分级标准

### 🔴 致命（Critical）

**判定标准**：不修会导致**数据损坏、安全漏洞、系统崩溃、资金损失**。不是"可能出问题"，是"一定会出事"。

典型：
- SQL 注入 / XSS
- 并发下数据丢失 / 覆盖
- 金额用 double 计算
- 异常吞没导致错误被隐藏，引发连锁故障
- 资源泄漏在长时间运行后必崩

### 🟠 严重（Major）

**判定标准**：特定条件下会出问题——高并发、大数据量、异常输入、边界场景。当前能跑，但是定时炸弹。

典型：
- N+1 查询（数据量大时慢）
- 缺失边界条件处理（空值/极值时报错）
- 状态流转不完备（非法状态未拦截）
- 线程安全集合用错（低并发没事，高并发出问题）

### 🟡 建议（Suggestion）

**判定标准**：不影响功能正确性，但影响可读性/可维护性/性能上限。改了更好，不改不会出事。

典型：
- 命名不清晰
- 方法过长、嵌套过深
- 重复代码
- 魔法数字
- 注释是废话或缺失

### 🟢 肯定（Praise）

**判定标准**：代码中值得学习的**好实践**。

典型：
- 防御性编程到位（前置校验、空值处理）
- 优雅的设计模式运用
- 清晰的注释解释了"为什么"
- 合理的抽象和命名

> ⚠️ **必须有肯定部分**。审查不是挑刺大会。好的代码要表扬，建立正向反馈。哪怕只有一条也要写。

## 单条问题标准格式

```
**[级别图标] 问题标题**

- 📍 **位置**：`文件名:行号`（或"第 X-Y 行"）
- 🏷️ **类型**：代码BUG（C编号）/ 逻辑BUG（L编号）/ 改进（G编号）/ 语言专项（J/V/P编号）
- 📝 **描述**：问题是什么。客观陈述，对事不对人。
- 🔍 **根因**：为什么会这样。帮作者理解，而非教训。
- 💡 **建议**：怎么改。用"可以考虑…"。改进建议才给，致命/严重问题给方向。
```

### 示例：代码 BUG

```markdown
**🔴 并发下订单状态可能丢失**

- 📍 **位置**：`OrderService.java:45`
- 🏷️ **类型**：代码BUG（C3 并发问题）
- 📝 **描述**：`checkAndUpdateStatus` 方法先查后改，中间无锁，多线程并发调用时可能两个线程都查到旧状态，都执行更新，导致状态只推进了一步而非两步。
- 🔍 **根因**：典型的 TOCTOU（检查-执行竞态）。查询和更新不在同一原子操作内。
- 💡 **建议**：可以考虑用乐观锁（version 字段 + UPDATE WHERE version=?）或数据库行锁（SELECT ... FOR UPDATE）保证原子性。
```

### 示例：逻辑 BUG（疑似）

```markdown
**❓ 退款金额计算逻辑待确认**

- 📍 **位置**：`RefundService.java:78`
- 🏷️ **类型**：逻辑BUG 疑似（L4 金额计算）
- 📝 **描述**：退款金额 = 订单金额 × 数量，但订单有折扣。当前逻辑退的是折扣前金额还是折后金额，代码看不明确。
- 🔍 **根因**：变量名 `orderAmount` 语义不明，可能是原价也可能是实付。无业务文档确认。
- ⚠️ **待确认**：请确认退款规则是"按原价退"还是"按实付退"。如按实付，当前逻辑可能有误。
- 📌 **背景来源**：无业务文档，纯代码推断。
```

### 示例：改进建议

```markdown
**🟡 方法过长，建议拆分**

- 📍 **位置**：`ReportGenerator.java:120-185`
- 🏷️ **类型**：改进（G2 方法过长）
- 📝 **描述**：`generateReport` 方法 65 行，承担了取数、格式化、写文件三个职责。
- 💡 **建议**：可以考虑拆成 `fetchData` / `format` / `writeToFile` 三个方法，每个职责单一，也便于单独测试。
```

### 示例：值得肯定

```markdown
**🟢 前置参数校验做得好**

- 📍 **位置**：`OrderService.java:30-38`
- 🏷️ **类型**：值得肯定
- 📝 **描述**：方法开头用卫语句逐个校验了入参（订单号非空、数量为正、用户存在），非法输入提前 return 并抛业务异常。防御性编程到位，后续逻辑可以放心处理合法数据。继续保持。
```

## 审查概要怎么写

报告开头必须有一段概要，让用户 10 秒内掌握全貌：

```markdown
## 审查概要

- **审查对象**：`OrderService.java`（创建订单方法，约 80 行）
- **审查维度**：代码 BUG（全量）+ 逻辑 BUG（基于项目文档）
- **业务背景来源**：项目 `docs/order-flow.md` 订单流程文档
- **发现问题**：🔴 致命 1 ｜ 🟠 严重 2 ｜ 🟡 建议 3 ｜ 🟢 肯定 1 ｜ ❓ 疑似 1
- **一句话结论**：核心并发问题必须修，修完后逻辑基本可用，建议补一下边界处理。
```

---

## 报告尾部的审查范围说明

```markdown
## 审查范围说明

- ✅ 已审查：`OrderService.java` 第 20-100 行（createOrder 方法）
- ⏭️ 未审查：该文件其他方法、被调用的 `OrderMapper`、`PaymentService`
- 📌 逻辑审查基于：`docs/order-flow.md`（订单状态流转图）。如业务规则与此文档不一致，相关结论需复核。
```

## 措辞规范

| 场景 | ✅ 推荐 | ❌ 避免 |
|------|---------|---------|
| 描述问题 | "这段循环里每次都查 DB" | "你写得不对" |
| 给建议 | "可以考虑用批量查询优化" | "必须改成批量查询" |
| 根因解释 | "这里并发下会竞态" | "这是个低级错误" |
| 不确定 | "疑似有问题，待确认…" | "这里肯定是 BUG" |
| 肯定 | "这里的校验很到位" | （不写肯定，只挑刺） |

> 核心心法：**报告是帮助作者改进的工具，不是展示审查者聪明的舞台。**
