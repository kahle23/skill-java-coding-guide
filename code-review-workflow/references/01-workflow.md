# 01 — 工作流：从代码到审查报告

本工作流分 3 个阶段、3 道门禁（Gate）。**每个阶段的 Gate 检查通过后才能进入下一阶段。**

核心原则：**只审不改，分级报告**。审查输出报告，修改决定权在用户。

---

## 阶段 1：收集输入

**目标**：拿到要审查的代码，明确审查范围。

**动作**：

| 输入类型 | 说明 | 必需性 |
|---------|------|--------|
| 代码（文件路径 / 代码段） | 要审查哪些代码 | ⚠️必需，必须用户明确指定 |
| 审查重点（可选） | 用户特别关注的方面：性能？安全？逻辑？ | 可选，不指定则全面审 |
| 业务背景（可选） | 这个功能干什么、有什么业务规则 | 可选，没有则阶段 2 自动检索（见 04） |

> ⚠️ **必须用户指定审查范围**。不要主动扫描整个项目、不要猜测用户想审哪段代码。用户说"帮我审一下"但没有给代码 → 必须问"审哪个文件 / 哪段代码？"。

### Gate 1：输入充足确认

**过门检查清单**：

- [ ] 用户明确指定了要审查的代码（文件路径或代码段）
- [ ] 知道代码用什么语言（能从文件后缀或代码内容判断）

**没过 Gate 1 的典型情况**：

- 用户只说"帮我 review 一下" → 问"review 哪段代码？给我文件路径或贴代码。"
- 用户说"审一下这个功能"但没给代码 → 问"这个功能涉及哪些文件？给我路径。"

> **草台班子场景**：用户甩来一段代码或一个文件路径，啥都没说。这时**不要追问审查重点**，默认全面审（代码 BUG + 逻辑 BUG 都过一遍），审完在报告里问用户是否要聚焦某方面。

---

## 阶段 2：审查

**目标**：对指定代码做全面审查，覆盖代码 BUG 和逻辑 BUG 两类。

**动作分两步走**：

### 步骤 2a：代码 BUG 审查（快，靠清单）

按 `02-checklist.md` 的**代码 BUG 专项清单**逐项排查。这部分与业务无关，靠技术功底和清单：

- 空指针 / 空引用
- 资源泄漏（未关闭的流/连接/文件句柄）
- 并发问题（非线程安全集合 / 竞态条件 / 锁粒度）
- SQL 注入 / XSS / 安全漏洞
- 异常吞没 / 错误处理缺失
- 类型安全 / 隐式转换问题
- 性能隐患（N+1 查询 / 循环内建对象 / 全表扫描）

**语言专项**：根据代码语言，额外过一遍对应专项（Java/Vue/Python，见 02）。

### 步骤 2b：逻辑 BUG 审查（慢，要背景）

逻辑 BUG 脱离业务背景无法判断对错。按 `04-logic-context.md` 获取背景：

1. **扫文档**：找项目的 `doc/`、`docs/`、`README.md`、需求文档等，提取业务规则。
2. **委托 AI 探测知识库技能**：让 AI agent 判断当前环境是否有已安装的项目知识库技能（如 taihong-knowledge 等），有则调用获取业务背景。
3. **代码推断**：从代码的命名、注释、相关调用链推断业务意图。

拿到背景后，按 `02-checklist.md` 的**逻辑 BUG 专项清单**审查：

- 边界条件（空值/极值/越界）
- 状态流转（状态机是否完备）
- 重复提交 / 幂等性
- 金额 / 数量计算精度
- 权限判断是否缺失
- 数据一致性（并发更新 / 事务边界）

> ⚠️ **背景不足时的降级**：如果文档没有、没有知识库技能、代码也推断不出业务规则，逻辑层**只标注"疑似"**，说明"无法确认此处逻辑是否正确，建议确认业务规则：……"。不要硬猜。

### Gate 2：审查完整性确认

**过门检查清单**：

- [ ] 代码 BUG 专项清单逐项过了（不是凭印象，是真的对着清单查）
- [ ] 语言专项过了（Java/Vue/Python 对应项）
- [ ] 逻辑 BUG 专项清单过了（或已标注"背景不足，疑似"）
- [ ] 每个发现的问题都记录了：位置（文件:行号）、类型、描述、根因
- [ ] 好的代码也记下来了（准备表扬）

**没过 Gate 2 的典型情况**：

- 只凭印象审了几个点，没对着清单 → 回去对着 02 的清单逐项过
- 逻辑 BUG 没查（因为没背景）→ 至少标注"背景不足"，列出"如果业务规则是 X，这里可能有问题"
- 代码量很大，审不完 → 先输出已审部分的报告，标注"未审部分：……"

---

## 阶段 3：输出报告

**目标**：按模板输出结构化报告，分级清晰，每条问题可执行。

**动作**：

按 `03-report-template.md` 输出报告。核心结构：

```
审查概要（审了什么、用了什么背景）
├── 🔴 致命问题（必须修，否则上线出事）
├── 🟠 严重问题（强烈建议修，有隐患）
├── 🟡 改进建议（可选优化，提升质量）
├── 🟢 值得肯定（好的实践，继续保持）
└── 疑似问题（背景不足无法确认，列出待确认点）
```

每条问题的标准格式：

```
**[级别] 问题标题**
- 📍 位置：文件名:行号
- 🏷️ 类型：代码BUG / 逻辑BUG / 改进 / 表扬
- 📝 描述：问题是什么
- 🔍 根因：为什么会这样
- 💡 建议：怎么改（可选，改进建议才给）
```

### Gate 3：报告质量检查

**过门检查清单**：

- [ ] 分级准确（致命/严重/建议/表扬 没放错层级）
- [ ] 每条问题都有位置（文件:行号）、类型、描述、根因
- [ ] 逻辑 BUG 的背景来源说明了（从哪得知的业务规则）
- [ ] 疑似问题明确标注了"疑似"和待确认点
- [ ] 改进建议是"建议"不是"命令"（用了"可以考虑"而非"必须"）
- [ ] 有值得肯定的部分（不只挑毛病）
- [ ] 报告开头有审查概要（审了什么代码、用了什么背景、发现几类问题各多少）

---

## 草台班子速查

| 情况 | 处理 |
|------|------|
| 用户只甩代码没说重点 | 默认全面审（代码+逻辑），审完问是否要聚焦 |
| 没有业务文档也没有知识库技能 | 逻辑层只标注"疑似"+待确认点，不硬猜 |
| 代码量很大 | 分块审，先输出已审部分报告，标注未审部分 |
| 用户说"快速看看" | 只过代码 BUG 专项（快），跳过逻辑 BUG，报告里说明"未做逻辑审查" |
| 发现好代码 | 记下来，报告里表扬（建立正向反馈） |
| 不确定是不是 BUG | 标注"疑似"，说明依据和待确认点，不装确定 |
| 审查重点与用户关注不符 | 报告输出后问"是否要聚焦 XX 方面深入审？" |
