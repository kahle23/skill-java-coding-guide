# 代码审查指南

通用代码审查技能，**语言无关**（Java / Vue / Python / 其他）。不绑定特定框架，按方法论审查，不凭感觉。

## 本技能性质

这是 **Workflow 型技能**（流程引导）。核心动作是**端到端跟做**——引导你走完从收集输入到输出报告的完整流程，每个阶段有过门检查（Gate），过门才进下一步。与 Reference 型技能（查规范）不同，本技能不需要从头读，跟着流程走就行。

## 两类 BUG，两种打法

代码审���要分清两类问题，打法完全不同：

| 类型 | 特征 | 打法 |
|------|------|------|
| **代码 BUG** | 技术层面的缺陷：空指针、资源泄漏、并发问题、SQL 注入、类型错误、异常吞没等。**与业务无关，换任何项目都成立。** | 凭技术功底直接审，靠审查清单逐项排查（见 02）。不需要业务背景。 |
| **逻辑 BUG** | 业务层面的错误：数据流转错、状态流转错、边界条件漏、权限判断缺失、重复提交未防、金额计算精度丢失等。**脱离业务背景无法判断对错。** | ⚠️ 必须先获取业务背景（见 04），否则只能审代码层面、逻辑层只能标"疑似"。 |

> 一段代码可能同时有两类问题。流程上先过代码 BUG（快，靠清单），再过逻辑 BUG（慢，要背景）。

## 流程总览

```
阶段1 收集输入 → Gate1 → 阶段2 审查（代码BUG + 逻辑BUG）→ Gate2 → 阶段3 输出报告 → Gate3
```

每个 Gate 都有过门检查清单，详见 `01-workflow.md`。

## 审查原则（讲究方式方法）

代码审查不是找茬，是帮助作者改进代码。遵循以下原则：

1. **对事不对人** — 问题描述指向代码，不指向作者。"这段循环里有 N+1 查询"而非"你写得不对"。
2. **说清为什么** — 每个问题都要解释根因和后果，让作者理解而非盲从。"这里线程不安全，并发下 Map 可能丢数据"而非"这里不对"。
3. **给方案不替做** — 提出改进建议，但修改决定权在作者（用户）。建议用"可以考虑…"而非"必须改成…"。
4. **分级说话** — 不是所有问题都一样重要。致命/严重/建议/表扬，区别对待（见 03）。
5. **不确定就标注** — 拿不准是不是 BUG（尤其逻辑层），标注"疑似"，说明依据和需要确认的点，不装确定。
6. **先肯定再建议** — 好代码要表扬，不只挑毛病。建立正向反馈。

## 凭证与安全红线

- **不改代码**：审查只输出报告，不直接修改用户代码（用户看完报告自己决定怎么改）。
- **不收集敏感信息**：不读取 `.env`、`application.yml` 中的明文密码、密钥配置。遇到含敏感信息的代码，提醒用户脱敏，不在报告中回显敏感值。
- **背景检索只读**：扫文档目录、查知识库技能时，只读不写，不改项目任何文件。
- **委托 AI 探测技能**：逻辑背景检索时，不硬编码调用某个特定技能（如 taihong-knowledge），而是让 AI agent 自行判断当前环境有哪些已安装的技能可用，有则用、无则降级（见 04）。

## 关联技能

- `java-coding-standard` — Java 编码规范，审查 Java 代码时代码 BUG 专项可参考其异常处理（03）、并发（07）、API 设计（08）章节。
- `python-coding-standard` — Python 编码规范，审查 Python 代码时代码 BUG 专项可参考其异常处理、类型提示章节。
- `db-design-standard` — 表结构规范，审查涉及数据库操作的代码时，辅助判断 SQL/索引是否合理。
- `code-to-doc-workflow` — 代码转文档。若审查发现文档严重缺失导致难以判断逻辑，可建议用户先用该技能补文档。
- 各项目知识库技能（如有）— 逻辑 BUG 审查时，委托 AI agent 探测并调用，获取业务背景。技能名不固定，按项目实际安装情况而定。
